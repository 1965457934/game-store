## 1. 算法概览

本系统包含两个层次的推荐：

| 推荐场景 | 算法类型 | 个性化程度 | 核心特征 | 调用接口 |
|---------|---------|-----------|---------|---------|
| **首页热门游戏** | 基于内容的加权随机 | 无（全站统一） | 销量 + 评分 + 时间衰减 | `GET /game/top` |
| **购物车猜你喜欢** | 基于用户行为的协同过滤（简化版） | 强（千人千面） | 用户分类偏好 + 销量 + 评分 | `GET /game/recommend` |

两者的共性：
- 均采用 **轮盘赌加权随机（Roulette Wheel Selection）** 作为最终抽样手段，保证高分候选优先曝光，同时避免结果固化。
- 均使用 **销量 + 评分** 作为游戏质量的基础过滤器，防止低质游戏进入推荐池。

## 2. 首页热门游戏推荐算法

### 2.1 设计目标

解决纯销量排序导致的**"老游戏霸榜"**问题：
> 一款上架一年的经典游戏销量 5w，一款上周上架的新游销量 1w。  
> 若单纯按销量排序，新游永远排不进前 6。  
> 但站在运营角度，**新上架的优质游戏应获得更高的曝光权重**。

因此引入**时间衰减因子**，让"新鲜度"成为与销量、评分并列的第三维度。

### 2.2 算法流程

```
┌─────────────────────────────────────────────────────────────────┐
│  Step 1: 候选池构建                                              │
│  从数据库取销量前 200 的上架游戏。limit 通常为 6，取 200          │
│  是为了在足够大的池子里做随机，兼顾性能与多样性。                  │
├─────────────────────────────────────────────────────────────────┤
│  Step 2: 三维评分计算                                             │
│  对每款游戏计算综合得分：                                          │
│  score = salesScore × ratingScore × timeDecay                   │
├─────────────────────────────────────────────────────────────────┤
│  Step 3: 轮盘赌加权随机抽样                                        │
│  按 score 为权重，不重复地随机抽出 limit 个游戏。                  │
│  概率与得分正相关，保证多样性。                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 三维评分模型

#### 维度一：销量分（`salesScore`）

```java
double sales = game.getSales() != null ? game.getSales().doubleValue() : 0;
double salesScore = Math.max(sales, 1);
```

| 参数 | 说明 |
|------|------|
| `sales` | 数据库中该游戏的累计销量 |
| `Math.max(sales, 1)` | 销量保底为 1，防止 0 销量游戏永远无法被选中 |

**设计理由**：销量是市场对游戏最直接的认可指标。保底处理确保即使是全新上架、尚未产生销量的游戏，也有基础得分参与竞争。

---

#### 维度二：评分分（`ratingScore`）

```java
double rating = game.getRating() != null ? game.getRating() : 3.0;
double ratingScore = rating / 5.0;
```

| 评分 | 系数 | 说明 |
|------|------|------|
| 5.0 | 1.00 | 满分，完全保留销量权重 |
| 4.5 | 0.90 | 优秀 |
| 4.0 | 0.80 | 良好 |
| 3.0 | 0.60 | 默认（无评分时） |
| 2.0 | 0.40 | 较差 |
| 1.0 | 0.20 | 很差 |

**设计理由**：将 0~5 分的绝对评分映射到 0~1 的相对系数，直接参与乘法运算。默认 3.0 分（0.6）避免无评分的新游戏被过度惩罚。

---

#### 维度三：时间衰减分（`timeDecay`）—— 本算法核心创新点

```java
long daysSinceRelease = ChronoUnit.DAYS.between(game.getCreateTime(), now);
double timeDecay = 30.0 / (30.0 + daysSinceRelease);
```

这是一个**倒数衰减模型**。

| 上架天数 | 衰减系数 | 直观含义 |
|---------|---------|---------|
| 0 天（今天上架） | 1.00 | 满分新鲜度 |
| 7 天 | 0.81 | 几乎无衰减 |
| 15 天 | 0.67 | 轻微衰减 |
| 30 天 | 0.50 | 衰减一半 |
| 60 天 | 0.33 | 衰减至 1/3 |
| 90 天 | 0.25 | 衰减至 1/4 |
| 180 天 | 0.14 | 大幅衰减 |

**为什么选择倒数衰减而非线性衰减？**

```
线性衰减：decay = 1 - days/90      → 90 天后直接归零（过于激进）
倒数衰减：decay = 30/(30+days)     → 永不为零，且前期平滑（更合理）
```

倒数衰减的优势：
1. **前期保护**：新游戏在前两周衰减很小（>0.8），确保有充足冷启动曝光期。
2. **长期存活**：老游戏不会归零，经典作品仍有机会被推荐。
3. **数学优雅**：曲线平滑，无需设置截断阈值。

---

#### 综合得分示例

| 游戏 | 销量 | 评分 | 上架天数 | 销量分 | 评分分 | 衰减分 | **综合得分** |
|------|------|------|---------|--------|--------|--------|-------------|
| A（新游） | 10,000 | 4.5 | 7 | 10,000 | 0.90 | 0.81 | **7,290** |
| B（老游） | 10,000 | 4.5 | 30 | 10,000 | 0.90 | 0.50 | **4,500** |
| C（爆款老游） | 50,000 | 4.0 | 90 | 50,000 | 0.80 | 0.25 | **10,000** |

**解读**：
- A 与 B 销量、评分完全相同，但 A 因更新鲜，得分是 B 的 **1.62 倍**。
- C 虽然是 90 天前的老游戏，但凭借 5w 销量和 4.0 评分，仍获得最高得分，说明销量规模可以部分抵消时间衰减。

### 2.4 轮盘赌加权随机

#### 为什么需要随机？

如果直接按 `score` 排序取前 6，每次刷新首页都展示同样的 6 款游戏，会导致：
- **马太效应**：头部游戏越来越强，腰尾部永无出头之日。
- **审美疲劳**：用户每次进来看到的都一样，缺乏新鲜感。

轮盘赌算法在"保证高分优先"与"提供多样性"之间取得平衡。

#### 算法伪代码

```
function rouletteWheelSelection(pool, limit):
    result = []
    for i from 0 to limit-1:
        if pool is empty: break
        
        totalScore = sum(score for (_, score) in pool)
        r = random(0, totalScore)   // 在轮盘上投掷飞镖
        accum = 0
        
        for (game, score) in pool:
            accum += score          // 轮盘转动
            if accum >= r:          // 飞镖落在此区域
                result.add(game)
                pool.remove(game)   // 不移除会导致重复
                break
    return result
```

#### 概率计算示例

假设候选池只有 A、B、C 三款游戏，得分分别为 7290、4500、10000。

```
总得分 = 7290 + 4500 + 10000 = 21790

P(A被选中) = 7290 / 21790 ≈ 33.5%
P(B被选中) = 4500 / 21790 ≈ 20.6%
P(C被选中) = 10000 / 21790 ≈ 45.9%
```

C 概率最高，但 A 仍有 1/3 的机会，B 也有 1/5 的机会。

### 2.5 公式速查

```
score = max(sales, 1) × (rating / 5.0) × (30 / (30 + days))

选中概率 = score_i / Σ(score)
```

## 3. 购物车猜你喜欢推荐算法

### 3.1 设计目标

在用户即将结账的场景下，推荐**与用户历史偏好相关**的游戏，提升客单价和转化率。

> 用户买了 3 款 RPG、2 款 FPS → 系统应优先推荐 RPG 和 FPS，而非策略或竞速。

### 3.2 与热门推荐的差异

| 对比维度 | 首页热门推荐 | 购物车猜你喜欢 |
|---------|-------------|---------------|
| **数据来源** | 全局销量排序 | 用户个人订单历史 |
| **核心特征** | 销量 + 评分 + 时间衰减 | 分类权重 + 销量 + 评分 |
| **个性化** | 无，所有用户看到的一样 | 强，千人千面 |
| **排除已购买** | 否 | **是** |
| **应用场景** | 首页引流 | 购物车 upsell |

### 3.3 算法流程

```
┌─────────────────────────────────────────────────────────────────┐
│  Step 1: 用户偏好提取（分类权重计算）                             │
│  统计用户所有已支付订单中，各分类的购买频次。                      │
│  例：RPG 3次，FPS 2次，策略 1次                                   │
│  归一化：RPG 50%，FPS 33.3%，策略 16.7%                           │
├─────────────────────────────────────────────────────────────────┤
│  Step 2: 候选池构建（排除已购买）                                 │
│  取所有上架游戏，LEFT JOIN 用户已购买列表，过滤 IS NULL 的部分。    │
│  避免"已经买了还推荐"的负面体验。                                 │
├─────────────────────────────────────────────────────────────────┤
│  Step 3: 个性化得分计算                                          │
│  score = categoryWeight × (sales + 1) × (rating / 5.0)          │
├─────────────────────────────────────────────────────────────────┤
│  Step 4: 轮盘赌加权随机抽样                                       │
│  同热门推荐算法，保证偏好类型优先，但结果有变化。                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 分类权重计算

#### SQL 查询逻辑

```sql
SELECT g.category_id AS categoryId, COUNT(*) AS count
FROM order_info o
JOIN order_item oi ON o.id = oi.order_id
JOIN game g ON oi.game_id = g.id
WHERE o.user_id = #{userId}
  AND o.status = 1        -- 仅统计已支付订单，未支付/取消的不算
GROUP BY g.category_id;
```

**为什么只统计 `status = 1`（已支付）？**
- 未支付订单可能只是用户的临时兴趣，不代表真实偏好。
- 已支付订单是用户用真金白银投票的结果，最能反映真实喜好。

#### 归一化计算

```java
// 原始计数
{ RPG: 3, FPS: 2, 策略: 1 }
totalWeight = 3 + 2 + 1 = 6

// 归一化后（概率分布）
{ RPG: 3/6=0.5, FPS: 2/6≈0.333, 策略: 1/6≈0.167 }
```

归一化的意义：将绝对计数转为相对概率，使不同用户之间可比，且便于后续乘法运算。

### 3.5 个性化得分模型

```java
double score = categoryWeight × (sales + 1) × (rating / 5.0);
```

#### 与热门推荐得分的区别

热门推荐的 `timeDecay`（时间衰减）被替换为 `categoryWeight`（分类权重）。

原因：
- 首页热门需要**全局公平性**——新游戏应被额外扶持。
- 购物车猜你喜欢需要**个体相关性**——用户爱玩的类型应被优先推荐。

#### 对未购买分类的处理

```java
double weight = categoryWeightMap.getOrDefault(game.getCategoryId(), 0.005);
```

用户未购买过的分类，默认给 **0.005** 的极小权重。

**为什么不给 0？**
- 若给 0，该分类游戏将完全无法被选中，推荐结果过于单调。
- 给极小值保留了"探索性推荐"的可能性，偶尔给用户一些惊喜。

#### 得分计算示例

假设用户偏好：RPG 权重 0.5，FPS 权重 0.333

| 游戏 | 分类 | 分类权重 | 销量 | 评分 | 计算过程 | 得分 |
|------|------|---------|------|------|---------|------|
| A | RPG | 0.50 | 5,000 | 4.5 | 0.50 × 5001 × 0.90 | **2,250.45** |
| B | FPS | 0.333 | 8,000 | 4.0 | 0.333 × 8001 × 0.80 | **2,131.47** |
| C | 策略 | 0.005 | 3,000 | 4.8 | 0.005 × 3001 × 0.96 | **14.40** |

**解读**：
- A 虽然销量低于 B，但用户更偏好 RPG，因此 A 的得分略高于 B。
- C 属于用户从未购买过的策略类，即使评分很高，也因权重极小而被大幅降权。

### 3.6 降级策略

系统在任何异常情况下都保证有推荐结果返回，不会让用户看到空白。

| 触发条件 | 降级行为 | 原因 |
|---------|---------|------|
| `userId == null`（未登录） | 调用 `getTopGames(limit)` | 无用户画像，只能推全局热门 |
| 用户无已支付订单 | 调用 `getTopGames(limit)` | 无行为数据，无法做个性化 |
| 权重计算后全为 0 | 调用 `getTopGames(limit)` | 数据异常保护 |
| 候选池为空（所有游戏都买过了） | 调用 `getTopGames(limit)` | 避免返回空列表 |

## 4. 前端联动说明

### 首页热门推荐

```vue
<!-- Home.vue -->
const loadTopGames = async () => {
  const res = await getTopGames(6)   // GET /game/top?limit=6
  topGames.value = res || []
}
```

每次进入首页，API 返回的 6 款游戏可能不同（加权随机）。

### 购物车猜你喜欢

```vue
<!-- Cart.vue -->
const loadRecommendGames = async () => {
  const res = await getRecommendGames(4)   // GET /game/recommend?limit=4
  recommendGames.value = res || []
}
```

由于用户购买历史会变化，且存在加权随机，每次进入购物车看到的推荐也会变化。

## 5. 调优建议

### 热门推荐调优

| 调优方向 | 当前值 | 建议调整 |
|---------|--------|---------|
| 候选池大小 | 200 | 若游戏总量 < 500 可降为 100 提升性能；若 > 2000 可升至 500 提升多样性 |
| 时间衰减基数 | 30 | 若希望新游戏获得更大扶持，可降为 15；若希望老经典保留更多权重，可升为 60 |
| 评分默认值 | 3.0 | 若平台整体评分偏高，可升为 3.5；若偏低，可降为 2.5 |

### 猜你喜欢调优

| 调优方向 | 当前值 | 建议调整 |
|---------|--------|---------|
| 未购买分类兜底权重 | 0.005 | 若希望增加"探索性"，可升为 0.02；若希望严格聚焦偏好，可降为 0.001 |
| 仅排除已购买 | - | 可扩展为"排除已购买 + 已加入购物车"，避免重复推荐 |
| 时间衰减引入 | 无 | 可考虑加入时间衰减，让用户近期购买的行为权重更高 |

### 未来扩展方向

1. **引入协同过滤**：找到"购买了游戏 A 的用户也购买了游戏 B"的关联规则。
2. **引入用户画像标签**：除分类外，加入价格敏感度（常买打折/原价）、游戏时长偏好等维度。
3. **A/B 测试框架**：对不同的权重参数分组实验，用点击率、转化率验证最优配置。
4. **冷启动优化**：新用户前 3 次访问用全局热门 + 随机探索，积累足够数据后再切换个性化推荐。
